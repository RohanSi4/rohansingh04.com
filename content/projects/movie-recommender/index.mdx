## what I wanted

I like recommendation systems, but I did not want this project to end as another
model trapped in a notebook. I wanted to search for a movie, get picks that made
sense, and understand why each title showed up.

## what I built

I built the full path from raw ratings to a site someone can use:

- recommendations for a MovieLens user based on historical ratings
- title search that finds similar movies from a selected starting point
- TMDB metadata for genres, release year, runtime, popularity, and posters
- a Next.js interface backed by a low-latency Go service

![Searching for movies in the recommender interface](/projects/movie-recommender-ui.jpg)

![Ranked movie recommendations with scores and reasons](/projects/movie-recommender-results.jpg)

The system has three parts. Python loads MovieLens data, adds TMDB metadata,
builds features, and evaluates a LightGBM ranking model. A Go API handles the
live recommendation path. Next.js turns those results into something easy to
search and compare.

## what was difficult

Model quality was only half the problem. I also had to decide how training data,
exported features, ranking endpoints, and the interface should fit together so a
recommendation could arrive quickly and still have a useful explanation.

The offline LightGBM model beat the baseline by 11.8%. The current Go service
uses a lighter heuristic over the exported features, which keeps the public demo
fast while full online model inference remains the next serving step. I would
rather make that tradeoff clear than pretend the demo is serving a model it is
not serving yet.

## what works now

The [live app](https://movie-reccomender-system-red.vercel.app/) supports title
search and user-based recommendations with posters, scores, and short reasons.
The current Go path responds in about 17 milliseconds. The offline evaluation
stays separate so anyone can rerun it and check the quality claim.
